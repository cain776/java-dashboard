## 1. 기획 배경 및 목적

### 1.1 배경

서비스기획팀은 홈페이지·앱·비젼&인사이트 3개 영역의 서비스를 기획·운영하고 있다.
그러나 팀의 업무 성격이 **테크니컬 지원·안정화 유지** 중심이다 보니, KPI를 수치화하기 어려운 구조적 한계가 있었다.

기존 방식대로 "요청 대응 SLA", "버그 처리율" 같은 **수동적(Reactive) 지표**만 사용하면:

- 잘해도 티가 나지 않고, 못하면 욕먹는 구조
- 경영진 평가에서 "그래서 뭘 했는데?"로 저평가
- 팀이 "티켓 처리반"으로 포지셔닝되는 리스크

### 1.2 목적

KPI 관점을 **수동형 → 능동형(Proactive)**으로 전환하여, 서비스기획팀이 **병원에 실질적으로 기여하는 가치**를 숫자로 증명할 수 있는 체계를 수립한다.

### 1.3 관점 전환 원칙

| 기존 (수동형) | 변경 (능동형) | 병원 기여 관점 |
|---|---|---|
| 요청 대응 SLA | 개선 과제 자체 발굴 건수 | **신환자 유입 증가** |
| 버그 수정 처리율 | 사전 리스크 제거율 | **환자 서비스 만족도 향상** |
| 데이터 요청 처리 | 시스템 데이터 활용·신뢰율 | **데이터 기반 경영 판단** |
| 장애 대응 시간 | 통계 화면 구축·안착률 | **현장 수기 업무 제거** |

**핵심 키워드**: 발굴 · 제안 · 주도 · 예방 · 임팩트

## 2. KPI 정의 — 4개 영역

### 2.1 홈페이지

| 항목 | 내용 |
|---|---|
| **부서** | 서비스기획팀 |
| **KPI명** | 홈페이지 개선 과제 **자체 발굴·릴리즈율** |
| **KPI 구분** | 능동 기여 KPI (Proactive Impact) |
| **정의** | 타 부서 요청이 아닌 **기획팀이 스스로 발굴·제안**하여 실제 릴리즈까지 이어진 개선 과제의 수와 비중 |
| **산식** | 자체 발굴률 = (기획팀 발굴 릴리즈 건수 ÷ 전체 릴리즈 건수) × 100<br>목표: 분기별 **자체 발굴 과제 최소 N건 릴리즈** |
| **현재 집계 방식** | Jira/Notion에서 `origin: planning-team` 태그 기반 집계, 분기 단위 |
| **데이터 출처** | 이슈 트래커, 릴리즈 노트, GA4(개선 전후 지표 비교) |
| **중요하게 보는 이유** | 홈페이지는 **신환자가 병원을 처음 만나는 접점**. 기획팀이 전환 퍼널의 병목을 먼저 찾아 개선하면 **상담 신청·예약 건수가 직접 늘어나며, 이는 곧 신환자 유입 증가 = 매출 증가**로 이어짐. 마케팅비를 추가로 쓰지 않고도 유입 효율을 높이는 가장 비용효과적인 방법 |
| **그룹 공통 KPI 포함 필요 여부** | ○ 포함 |
| **포함 필요 사유** | 병원 그룹 전체의 디지털 채널 환자 유입 효율을 공통 관리할 필요 |
| **관리상 혼선/개선 필요 사항** | "자체 발굴"의 정의 모호 → **발굴 출처 태깅 규칙** 사전 확정 필요 (예: 데이터 분석 / 사용자 피드백 / 경쟁사 벤치마킹) |
| **비고** | 릴리즈 후 **개선 전/후 지표 비교 리포트** 의무화 권장. 예: "예약 버튼 위치 변경 → 상담신청 +12%" |

### 2.2 앱

| 항목 | 내용 |
|---|---|
| **부서** | 서비스기획팀 |
| **KPI명** | 앱 **사전 리스크 제거 & 릴리즈 품질 주도율** |
| **KPI 구분** | 능동 품질 KPI (Proactive Quality) |
| **정의** | 릴리즈 전 기획 단계에서 **기획팀이 먼저 식별·제거한 잠재 리스크 건수**와, 기획팀 주도 릴리즈의 버그 발생률 |
| **산식** | 사전 제거율 = (기획 검토 단계 발견 이슈 ÷ 전체 발견 이슈) × 100<br>주도 릴리즈 품질 = 기획팀 주도 기능의 릴리즈 후 30일 버그 건수 / 기능 수 |
| **현재 집계 방식** | 기획 리뷰 회의록 + Jira 이슈 생성 시점(pre-dev/post-dev) 태깅 |
| **데이터 출처** | Jira, Crashlytics, 스토어 리뷰 |
| **중요하게 보는 이유** | 앱은 **환자가 병원 서비스를 24시간 경험하는 채널**. 예약·결과조회·알림이 매끄럽게 작동할수록 **환자 만족도와 병원 신뢰도가 높아짐**. 기획 단계에서 리스크를 미리 잡으면 ① 환자의 디지털 서비스 만족도 향상 ② 원무과·콜센터 직원이 **본업(환자 응대)에 더 집중**할 수 있는 환경 조성 ③ 안정적 서비스 경험을 통한 **재방문·추천 의향 강화** — 앱 품질을 높이는 것이 곧 **"병원을 다시 찾고 싶게 만드는 힘"** |
| **그룹 공통 KPI 포함 필요 여부** | ○ 포함 |
| **포함 필요 사유** | 디지털 서비스 장애의 현장 파급력은 모든 병원 공통 이슈 |
| **관리상 혼선/개선 필요 사항** | 리스크 발견 시점 태깅이 없으면 측정 불가 → **pre-dev / post-dev 태그 필수화** |
| **비고** | 임팩트 환산 예시: "사전 제거 1건 = 현장 CS 대응 약 2시간 절감" 식으로 병원 운영비용으로 번역하면 경영진 보고에 효과적 |

### 2.3 비젼&인사이트

| 항목 | 내용 |
|---|---|
| **부서** | 서비스기획팀 |
| **KPI명** | 시스템 데이터 **활용·신뢰율** (Data Adoption Rate) |
| **KPI 구분** | 시스템 신뢰 KPI (System Trust & Adoption) |
| **정의** | 비젼&인사이트 시스템이 제공하는 데이터·리포트를 현장·경영진이 **실제 업무 판단에 활용하는 비율** — 시스템이 제공하는 데이터를 신뢰하고 채택하는 비율 |
| **산식** | 활용률 = (시스템 데이터 기반 의사결정 건수 ÷ 전체 의사결정 건수) × 100<br>보조: 대시보드 MAU, 리포트 조회율, 수기엑셀 대체율 |
| **현재 집계 방식** | 시스템 접속 로그 + 분기 설문(의사결정 시 참고 출처 조사) |
| **데이터 출처** | 시스템 로그, 사용자 설문, 경영회의록 |
| **중요하게 보는 이유** | 아무리 좋은 시스템을 만들어도 **현장이 안 쓰면 투자 대비 효과가 0**. 시스템 데이터 활용률이 높아질수록 ① 감 대신 근거 기반 운영 ② 수기 집계 시간 절감 ③ 데이터 기반 문화 정착 → **시스템이 병원의 의사결정 인프라로 자리잡는 것**이 곧 병원 경쟁력 |
| **그룹 공통 KPI 포함 필요 여부** | ○ 포함 |
| **포함 필요 사유** | 시스템 데이터 신뢰·활용 수준은 병원 그룹 디지털 성숙도의 핵심 지표 |
| **관리상 혼선/개선 필요 사항** | "활용했다"의 판정 기준 모호 → 시스템 로그(조회·다운로드) + 설문 병행으로 교차 검증 필요 |
| **비고** | 단계별 목표 권장: 1단계 "접속률 확보" → 2단계 "조회 습관화" → 3단계 "의사결정 기반화" |

### 2.4 통계 화면 구축

| 항목 | 내용 |
|---|---|
| **부서** | 서비스기획팀 |
| **KPI명** | 통계 화면 **기획·구축 완성도 및 현장 안착률** |
| **KPI 구분** | 능동 인프라 KPI (Proactive Infrastructure) |
| **정의** | 병원 운영에 필요한 통계 화면을 기획팀이 **주도적으로 설계·구축**하고, 현장이 수기 대신 **시스템 통계를 실제 업무에 활용하는 비율** |
| **산식** | 구축 완성도 = (기획 완료 통계 지표 수 ÷ 현장 요구 지표 수) × 100<br>현장 안착률 = (시스템 통계로 대체된 수기 보고 수 ÷ 전체 수기 보고 수) × 100<br>보조: 대시보드 MAU, 지표별 조회 빈도 |
| **현재 집계 방식** | 지표 정의서 기반 진행률 관리(분기), 시스템 접속 로그 + 현장 수기 현황 대장 비교(월간) |
| **데이터 출처** | 지표 정의서, 시스템 접속 로그, 수기 보고 현황 대장, 현장 설문 |
| **중요하게 보는 이유** | 현장은 아직 **엑셀·수기 대장으로 초진 수, 수술 전환율, 매출을 집계**하고 있음. 이 시간이 곧 인건비이며 수기는 오류가 잦음. 기획팀이 통계 화면을 설계·안착시키면 ① 현장 집계 업무 시간 절감 → **본업(환자 응대) 집중** ② 실시간 정확한 숫자로 **경영 판단 속도·정확도 향상** ③ 수기 오류로 인한 **잘못된 의사결정 방지** — 통계 화면은 병원이 **"느낌이 아닌 숫자로 돌아가는 조직"**이 되기 위한 기반 인프라이며, 이를 설계하는 것이 기획팀의 고유 역할 |
| **그룹 공통 KPI 포함 필요 여부** | ○ 포함 |
| **포함 필요 사유** | 데이터 인프라 구축 수준은 병원 그룹의 디지털 성숙도를 결정하는 기반 지표 |
| **관리상 혼선/개선 필요 사항** | ① "필요 지표"의 범위 확정 필요 — 현장 인터뷰 후 지표 목록 확정이 선행되어야 분모가 정해짐 ② 수기→시스템 전환 판정 기준 — "한 달간 수기 미사용"을 전환 완료로 볼 것인지 정의 필요 ③ 구축 완성도(기획)와 안착률(활용)은 시차가 있으므로 분기별 각각 별도 추적 |
| **비고** | 단계별 목표 권장: **Q1** 현장 인터뷰 + 지표 정의서 완성 → **Q2** 핵심 지표 5종 화면 기획·개발 협업 → **Q3** 파일럿 운영 + 피드백 반영 → **Q4** 전체 배포 + 수기 대체율 측정. 임팩트 환산 예시: "수기 일일보고 30분/일 → 시스템 자동화 시 연간 약 180시간 절감" |

## 3. 통계 화면 구축 — 기획팀 주도 상세 계획

### 3.1 전체 프로세스에서 기획팀의 역할

| 단계 | 기획팀 비중 | 핵심 활동 |
|---|---|---|
| ① 요구사항 수집 | 100% | 현장 인터뷰, 수기 현황 파악, 벤치마킹, 우선순위 도출 |
| ② 지표 설계 | 80% | 지표 정의서, 지표 간 관계 설계, 기준값 조사, 이상치 기준 |
| ③ 화면 기획 | 100% | 대시보드 구성안, 필터·드릴다운, 권한별 뷰, 용어 표준화 |
| ④ 개발 협업 | 30% | 데이터 매핑 확인, 스펙 전달·QA, 엣지케이스 정의 |
| ⑤ 검증·안착 | 70% | 파일럿 운영, 현장 교육, 활용 모니터링, 수기 대체 확인 |

### 3.2 단계별 산출물

#### ① 요구사항 수집

| 작업 | 상세 | 산출물 |
|---|---|---|
| 현장 인터뷰 | 원장·실장·원무·간호 등 직군별로 "어떤 숫자를 보고 싶은지" 수집 | 인터뷰 기록서 |
| 기존 수기 현황 파악 | 현재 엑셀·수기장부로 보고 있는 통계가 뭔지 전수 조사 | 수기 통계 목록표 |
| 경쟁사·벤치마킹 | 타 안과·병원 시스템의 통계 화면 사례 수집 | 벤치마킹 리포트 |
| 우선순위 도출 | 빈도(매일/주간/월간) × 중요도(의사결정 영향) 매트릭스로 순위 결정 | 우선순위 매트릭스 |

#### ② 지표 설계

| 작업 | 상세 | 산출물 |
|---|---|---|
| 지표 정의서 작성 | 지표명, 정의, 산식, 집계 주기, 데이터 출처를 명확히 정의 | 지표 정의서 |
| 지표 간 관계 설계 | 상위 지표 → 하위 지표 드릴다운 구조 설계 | 지표 트리맵 |
| 기준값(Baseline) 조사 | 현재 수치가 얼마인지 파악 → 목표치 설정 근거 | 기준값 보고서 |
| 이상치 기준 정의 | "이 숫자가 X 이하면 경고" 같은 알림 기준 | 알림 기준표 |

**지표 정의서 예시**:

| 지표명 | 정의 | 산식 | 집계 주기 | 데이터 출처 | 열람 대상 |
|---|---|---|---|---|---|
| 일 초진 환자 수 | 당일 최초 내원한 환자 수 | COUNT(첫 방문 환자) | 일간 | 예약DB + 접수DB | 원장, 실장 |
| 수술 전환율 | 상담 환자 중 수술 결정 비율 | (수술결정 ÷ 상담) × 100 | 주간 | 상담DB + 수술DB | 원장, 상담실장 |
| 재방문율 (30일) | 첫 방문 후 30일 내 재방문 비율 | (30일내 재방문 ÷ 초진) × 100 | 월간 | 접수DB | 경영진 |
| 예약 부도율 | 예약 후 미방문 비율 | (No-show ÷ 총예약) × 100 | 일간 | 예약DB | 원무, 콜센터 |
| 평균 대기 시간 | 접수~진료 시작 평균 소요시간 | AVG(진료시작 − 접수시각) | 일간 | 접수DB + 진료DB | 원무, 간호 |
| 상담사별 전환율 | 상담사별 수술 전환 비율 | (상담사 수술결정 ÷ 상담사 상담) × 100 | 월간 | 상담DB | 실장 |
| 채널별 유입 현황 | 홈페이지/앱/전화/방문 등 채널별 신환자 수 | COUNT(채널별 초진) | 주간 | 접수DB + GA4 | 마케팅, 경영진 |
| 매출 추이 | 진료과목별/기간별 매출 | SUM(수납액) GROUP BY 과목 | 일/주/월 | 수납DB | 경영진 |

#### ③ 화면 기획

| 작업 | 상세 | 산출물 |
|---|---|---|
| 대시보드 구성안 | 어떤 지표를 어떤 차트로 보여줄지 레이아웃 설계 | 와이어프레임 |
| 필터·드릴다운 설계 | 기간/부서/의사/채널 등 필터 조건과 클릭 시 상세 화면 흐름 | 화면 흐름도 |
| 권한별 뷰 설계 | 원장용 / 실장용 / 원무용 등 역할별로 보이는 지표 구분 | 권한 매트릭스 |
| 알림·리포트 설계 | 자동 리포트(일일/주간/월간) 발송 대상 및 포맷 | 리포트 스펙 |
| 용어 표준화 | "초진/재진", "상담/수술결정" 등 현장과 시스템 용어 통일 | 용어 사전 |

#### ④ 개발 협업

| 작업 | 기획팀 역할 | 개발팀 역할 |
|---|---|---|
| 데이터 매핑 확인 | 지표 산식에 필요한 테이블·컬럼이 있는지 확인 요청 | 실제 DB 구조 조사, 쿼리 작성 |
| 스펙 전달 & QA | 기획서 전달 → 개발 결과물이 스펙과 일치하는지 검증 | 화면 개발, API 구현 |
| 데이터 정합성 검증 | "이 숫자가 맞나?" 현장 확인 (수기 대장과 시스템 수치 비교) | 쿼리 보정 |
| 엣지케이스 정의 | 취소된 예약은 부도율에 포함? 당일 접수는 초진? 등 업무 규칙 정의 | 로직 반영 |

#### ⑤ 검증·안착

| 작업 | 상세 | 산출물 |
|---|---|---|
| 파일럿 운영 | 특정 부서에 2~4주 시범 사용 → 피드백 수집 | 파일럿 결과 보고서 |
| 현장 교육 | 대시보드 읽는 법, 필터 사용법, 숫자 해석법 안내 | 교육 자료, 1페이지 가이드 |
| 활용 모니터링 | 접속 로그 기반으로 실제 사용 여부 추적 | 월간 활용률 리포트 |
| 개선 사이클 | 현장 피드백 → 지표 추가/수정 → 재배포 반복 | 개선 이력 로그 |
| 수기 대체 확인 | 기존 엑셀·수기가 실제로 줄었는지 확인 | 수기→시스템 전환율 |

## 4. KPI 전체 요약

### 4.1 4종 KPI 한눈에 보기

| # | 영역 | KPI명 | KPI 구분 | 병원에 주는 가치 |
|---|---|---|---|---|
| 1 | 홈페이지 | 자체 발굴·릴리즈율 | 능동 기여 KPI | 마케팅비 추가 없이 **신환자 유입 증가** |
| 2 | 앱 | 사전 리스크 제거·품질 주도율 | 능동 품질 KPI | **환자 서비스 만족도 향상** → 재방문·추천 강화 |
| 3 | 비젼&인사이트 | 시스템 데이터 활용·신뢰율 | 시스템 신뢰 KPI | **감이 아닌 데이터로 판단하는 병원** |
| 4 | 통계 화면 | 기획·구축 완성도 및 현장 안착률 | 능동 인프라 KPI | **수기 집계를 없애고 현장이 본업에 집중** |

### 4.2 경영진 보고용 한 줄 요약

> **"기획팀은 환자를 더 데려오고, 서비스 품질을 높이고, 데이터로 판단하는 병원을 만들며, 그 모든 것을 숫자로 볼 수 있는 통계 체계를 설계하는 팀입니다."**

## 6. 참고 — KPI 구조 설계 시 유의사항

### 6.1 능동형 KPI를 잡을 때의 함정

| 함정 | 증상 | 방지책 |
|---|---|---|
| 양치기 | 발굴 건수만 늘리고 임팩트 없음 | **발굴 × 릴리즈 × 임팩트** 3단계 세트 측정 |
| SLA 품질 하락 | 기한 맞추려고 스펙 품질 저하 | **건수 + 품질 + 리드타임** 3개 세트 |
| 측정 불가 | 태깅 체계 없이 KPI만 선언 | **태깅 규칙 확정 → 2주 파일럿 → KPI 공식화** 순서 |

### 6.2 권장 KPI 비중 구조

```
① 안정화 KPI (20%) — 기본 운영 품질 유지 (기존 수동형 최소 유지)
② 능동 KPI  (40%) — 발굴·예방·제안 (이 기획서의 핵심)
③ 임팩트 KPI (40%) — 실제 병원 성과에 미친 영향
```

**임팩트 KPI를 40% 넣는 것이 핵심**. 이것이 없으면 기획팀은 여전히 "티켓 처리반"으로 인식된다.
