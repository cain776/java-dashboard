# 메뉴별 주간/월간 구성 가능성 정리

> 작성일: 2026-06-19
> 범위: **전체지표 · 예약 · 검사 · 전환&성공률 · 수술 · 외래** 그룹
> (마케팅 · 취소&부도 · 객단가 · 기타 그룹은 대부분 미구현이라 제외)
> 목적: 각 통계 메뉴를 **주간(주 단위)** 으로 분해할 수 있는지 데이터 소스 기준으로 판정하고,
> 이를 근거로 **주간 레포트** 구성/업데이트 방향을 정한다.

---

## 1. 판정 기준 — "주간으로 쪼갤 수 있는가"는 데이터 소스가 결정한다

각 메뉴의 백엔드 서비스 구현을 확인한 결과, 데이터 소스는 3가지로 나뉜다.

| 분류 | 의미 | 주간 분해 |
|------|------|----------|
| **(A) 라이브만** | 모든 연도를 운영 DB(MSSQL)에서 일자 컬럼으로 직접 집계 | ✅ **전 연도 주간 가능** |
| **(B) 혼합** | 2024·2025(일부는 2026 1~4월)는 **레거시 월별 확정값 하드코딩**, 그 외는 라이브 | ⚠️ **라이브 연도만 주간 가능** (과거 월 합계는 주로 못 쪼갬) |
| **(C) 상세 리스트** | 일자 단위 행을 그대로 나열, 기간은 단순 필터 | ✅ **주간/월간 모두 필터링 문제** |

> **핵심 함정**: 레거시 확정값(2024·2025)은 *월 합계*만 존재한다. 콜센터 MySQL·해피톡 등
> 다중 소스에서 산출된 뒤 폐기된 값이라 주 단위로 역산이 불가능하다. 따라서 (B) 유형 메뉴의
> 주간 뷰는 구조적으로 **"라이브 연도(2026~) 단일 연도"** 가 된다.

### 주간 집계의 신뢰성 주의 (공통)

- **시점 소급변동**: EXAM / OPERATIONDATA 등은 고객당 1행 *덮어쓰기* 구조라, 과거 주 수치가
  사후에 미세하게 바뀐다. 월 단위에선 묻히지만 **주 단위에선 변동이 더 잘 드러난다.**
  (참고: 검사 마감 스냅샷 기획 `docs/db/지표-마감스냅샷-기획.md`)
- **당주/미래 주**: 아직 데이터가 다 안 쌓여 급감 → 예약 종합처럼 `null` 처리 필요.
- **월 경계 처리**: 이미 구현된 `OverallExamWeeklyService`의 규칙을 표준으로 삼는다 →
  *1주 = [1일 ~ 첫 일요일], 이후 월~일 7일씩, 월 경계 주는 각 달로 클립하여 "월 합계 = 그 달 주 합계" 보장.*

### 요약

- **✅ 전 연도 주간 가능 (즉시 구현)** — 유입(검사예약), 예약 건수, 전환율, 예약률, 중단 사유,
  드림렌즈 검사건수, 각 리스트 3종
- **⚠️ 라이브 연도만 주간 가능** — 예약 종합, 검사건수, 시술별, 시력교정 검사건수, 외래수, 수술 3종
  → 주간 뷰는 2026~ 만, 과거 비교는 월간 유지
- **이미 구현됨** — 주간 검사자 종합지표 (월 경계 클립 로직의 레퍼런스)

## 3. 구현 방식 권장 (메뉴 추가 vs 화면 내 토글)

> 1차 논의 결론을 데이터 가용성 기준으로 일반화한 규칙:

- **✅ 전 연도 가능 메뉴** → **같은 화면 내 `주간 | 월간` 토글**.
  과거·현재 모두 동일하게 주로 쪼개지므로 토글 시 데이터가 비지 않는다. 메뉴 폭발 방지.
- **⚠️ 라이브만/부분 가능 메뉴** → **별도 "주간 ○○" 메뉴** (전체지표 패턴과 통일).
  토글로 묶으면 주간 전환 시 2024·2025 라인이 사라져 사용자가 혼란. 화면 캡션도 분리 가능
  ("월간=확정값 / 주간=라이브 추정치, 당주까지 신뢰").

근거 사례: `전체지표`는 이미 `월별 검사자 종합지표`(B, 월간) + `주간 검사자 종합지표`(A, 라이브)를
**별도 메뉴**로 분리해 두었다. 예약 종합·검사건수·외래수·수술도 같은 구조다.

## 4. 주간 레포트 업데이트 방향

현재 `주간 레포트`/`월간 레포트`는 동일한 `ReportPage` 골격(KPI 4종 + 섹션 4종)을
공유하는 **데이터 미연결 플레이스홀더**다 (`frontend/src/pages/ReportPage.tsx`).

### "월간 레포트를 주간 레포트로 다 바꿀 수 있는가?" → **조건부로 가능**

주간이 가능하면 월간은 *주의 합*으로 항상 도출되므로(월 경계 클립 설계), 기술적으로는
**기간 토글 하나로 두 레포트를 통합**할 수 있다. 다만 한 가지 제약이 있다:

| 항목 | 주간 레포트 | 월간 레포트 |
|------|------------|------------|
| 강점 | 최신 추세·당해연도 즉시성 | **당해연도·전년도·전전년도 3개년 비교** (현재 2024·2025·2026) |
| 라이브만 지표(예약종합 등) | 2026 단일 연도만 | 과거 확정값 비교 가능 |
| 결론 | 라이브 중심 운영 모니터링 | 과거 대비 성과 평가 |

→ **월간의 "3개년 비교" 가치는 주간으로 대체 불가**(과거 주간 원본 없음). 따라서:

1. **주간 레포트**는 **✅ 전 연도 가능 지표를 1순위로** 채운다
   (예약 건수 · 전환율 · 예약률 · 중단 사유 · 유입). 라이브만 지표(예약 종합·검사건수·외래수·수술)는
   **당해연도 주간**으로만 싣고, 과거 비교가 필요한 카드는 "월간 참조" 링크로 처리.
2. **월간 레포트**는 당분간 유지하되, 장기적으로 *기간 토글(주간/월간)을 가진 단일 레포트*로
   통합 가능. 단 토글이 '월간'일 때만 3개년 비교를 노출하고, '주간'일 때는 라이브 연도로 한정.
3. 두 레포트의 KPI 정의·계산은 위 판정표의 서비스를 그대로 재사용
   (레포트 전용 신규 집계 로직을 만들지 말 것 — DRY).

### 권장 액션 순서

1. 주간 레포트 섹션을 **✅ 전 연도 가능 지표**로 먼저 데이터 연결 (리스크 없음)
2. 라이브만 지표는 `OverallExamWeeklyService`의 주 버킷팅을 재사용해 **당해연도 주간** 추가
3. 월간 레포트는 기존대로 두고, 통합 토글은 주간 안정화 후 검토
