# 지표 마감 스냅샷 — 기획안 (주간 승인 + 월 마감 버전)

> 목적: 가변 소스 테이블(EXAM·RESERVATION 등)에서 나온 주간/월별 지표가 시간이 지나며 미세하게 변하는 문제(드리프트)를 없애고, **리포트 재현성**을 확보한다.
> 원칙: **주간 승인값을 쌓고, 월말에는 승인된 주간값을 검토해 월 확정값으로 닫는다.** 화면은 조회 1벌 + 마감관리 1벌로 시작한다.
> 최종 갱신: 2026-06-19 · 상태: 기획(미구현)

---

## 1. 배경

- `EXAM`은 고객당 1행 스냅샷 → 재검사 시 과거월 수치가 사후 변동.
- `RESERVATION` 상태값(`I/H`)은 늦은 내원·정정으로 과거월 행수가 사후 증가.
- 그래서 "그 순간엔 맞던 숫자"가 라이브 재계산 시 ±수 건 어긋난다(외래수 2026-01 레거시 4,822 vs 현재 4,831 등).
- 지금도 2024~2025는 **하드코딩(=수동 스냅샷)** 으로 고정 중. 이걸 테이블로 정식화하는 것이 이 기획의 핵심.

## 2. 핵심 원칙 (단 한 줄)

> **(지표, 기간) 승인/마감 스냅샷이 있으면 그 값을 쓰고, 없으면 라이브로 계산한다.**

- 마감된 과거월 → 스냅샷 존재 → **확정값** 표시 (다시는 안 변함)
- 승인된 과거주 → 주간 스냅샷 존재 → **승인값** 표시 (주간 리포트 재현)
- 당월/미마감월·미승인주 → 스냅샷 없음 → **라이브** 계산 (잠정값)

## 3. 테이블 (월 확정 1개 + 주간 승인 1개)

운영 앱 DB(H2 개발 / PostgreSQL 운영)에 저장한다. **CRM MSSQL은 READ-ONLY라 쓰지 않는다.**

### 3.1 월 마감 스냅샷

```sql
CREATE TABLE metric_monthly_snapshot (
  metric_key    VARCHAR(40)  NOT NULL,   -- 예: 'exam_count', 'outpatient_count', 'vision_exam', 'cataract_exam', 'dreamlens_exam'
  year          INT          NOT NULL,
  month         INT          NOT NULL,    -- 1~12
  value         INT,                      -- 확정 수치 (null 허용)
  rule_version  VARCHAR(20)  NOT NULL,    -- 산출 기준 버전 (예: 'v1')
  generated_at  TIMESTAMP    NOT NULL,    -- 확정 시각
  PRIMARY KEY (metric_key, year, month)
);
```

- 한 행 = 한 지표의 한 달 확정값.
- 월간 레포트·월별 통계 화면의 최종 기준값이다.

### 3.2 주간 승인 스냅샷

```sql
CREATE TABLE metric_weekly_snapshot (
  metric_key    VARCHAR(40)  NOT NULL,
  year          INT          NOT NULL,
  month         INT          NOT NULL,    -- 월 경계 기준 1~12
  week_index    INT          NOT NULL,    -- 해당 월의 1~6주차
  week_start    DATE         NOT NULL,
  week_end      DATE         NOT NULL,    -- 월 경계를 넘으면 해당 월 말일로 클립
  value         INT,
  rule_version  VARCHAR(20)  NOT NULL,
  approved_at   TIMESTAMP    NOT NULL,
  approved_by   VARCHAR(80)  NOT NULL,
  PRIMARY KEY (metric_key, year, month, week_index)
);
```

- 한 행 = 한 지표의 한 달 안 특정 주차 승인값.
- 주간 레포트와 `전체지표 > 주간 검사자 종합지표` 재현성을 확보한다.
- 월 마감 시에는 이 승인값을 기준으로 라이브 현재값과 차이를 보여주고, 최종 월 스냅샷을 저장한다.
- 단위·소스 등 메타는 코드/문서가 갖고 있으므로 테이블엔 최소한만 둔다.

## 4. 읽기 흐름 (서비스 공통)

```text
getWeekly(metricKey, weeks):
  for each week in 범위:
    snapshot = weeklySnapshotRepo.find(metricKey, week)
    value = snapshot != null ? snapshot.value           # 승인
                             : liveCompute(week)        # 잠정(라이브)
    status = snapshot != null ? 'approved' : 'provisional'

getMonthly(metricKey, years):
  for each (year, month) in 범위:
    snapshot = monthlySnapshotRepo.find(metricKey, year, month)
    value = snapshot != null ? snapshot.value          # 확정
                             : liveCompute(year, month) # 잠정(라이브)
    status = snapshot != null ? 'confirmed' : 'provisional'
```

- 응답 DTO에 `status` 한 필드만 추가한다. 값은 `provisional`, `approved`, `confirmed` 중 하나로 둔다.

## 5. 쓰기(마감) 흐름

가장 단순한 형태부터 시작하되, 운영 흐름은 **주간 승인 → 월 마감** 순서로 둔다.

1. **주간 승인 버튼/엔드포인트** — `POST /api/admin/metric-snapshot/approve-week?weekStart=2026-05-04&weekEnd=2026-05-10`
   - 해당 주의 모든 지표를 라이브 계산 → `metric_weekly_snapshot`에 upsert (`rule_version`, `approved_at`, `approved_by` 기록).
   - 운영자가 주간 회의/리포트 기준값을 확인한 뒤 "주간 승인" 누르면 그 주 값이 고정됨.
2. **월 마감 버튼/엔드포인트** — `POST /api/admin/metric-snapshot/close-month?ym=2026-05`
   - 해당 월의 모든 지표를 라이브 계산 → `metric_monthly_snapshot`에 upsert (`rule_version`, `generated_at` 기록).
   - 화면에서는 "승인된 주간 합계"와 "월 라이브 현재값" 차이를 먼저 보여준 뒤 관리자가 월 확정.
   - 운영자가 "이 달 확정" 누르면 월별 통계/월간 레포트 값이 고정됨.
3. (선택, 나중에) 스케줄 잡 — 매주 월요일 전주 후보 생성, 매월 1일 전월 마감 후보 생성.

> 1단계는 수동만으로 충분. 스케줄은 운영 안정화 후 추가.

### 5.1 스냅샷 풀기 (재마감 · 해제)

확정값이 틀렸거나 산출 규칙이 바뀌면 되돌릴 수 있어야 한다. 동작은 2가지뿐.

| 동작 | 방법 | 결과 |
|---|---|---|
| **주간 재승인(덮어쓰기)** | 같은 주 approve-week 다시 호출 | 라이브 재계산 → 같은 주간 행 upsert. 데이터 정정/규칙 변경 시 사용 |
| **월 재마감(덮어쓰기)** | 같은 달 close-month 다시 호출 | 라이브 재계산 → 같은 월간 행 upsert(`value`·`rule_version`·`generated_at` 갱신). 데이터 정정/규칙 변경 시 사용 |
| **해제(풀기)** | `DELETE /api/admin/metric-snapshot?ym=2026-05` | 스냅샷 행 삭제 → 그 달은 다시 **라이브(잠정)** 로 표시 |

- 둘 다 **관리자 수동**. 승인주/확정월을 영구 불변(immutable)으로 잠그지는 않는다 — 현실적으로 정정이 생기기 때문.
- 단순화를 위해 **이력은 남기지 않는다**(덮어쓰면 이전 값은 사라짐). 이력 추적이 필요해지면 그때 별도 테이블로 확장.
- `rule_version`이 바뀌는 전역 규칙 변경 시 = 해당 지표 전체 월을 재마감(일괄 close).

### 5.2 수정 흐름 (심플 버전)

수정은 별도 편집 기능을 만들지 않고 **풀기 → 다시 계산 → 다시 저장** 으로 처리한다.

```text
[승인/확정 상태]
       |
       +-- 풀기
       v
[잠정 상태: 라이브값 표시]
       |
       +-- 다시 계산
       v
[미리보기]
       |
       +-- 다시 저장
       v
[승인/확정 상태]
```

버튼은 아래 3개만 둔다.

| 버튼 | 동작 |
|---|---|
| 풀기 | 기존 주간 승인값/월 확정값을 삭제하거나 비활성화한다. 화면은 다시 라이브 잠정값을 보여준다. |
| 다시 계산 | 현재 운영 DB와 현재 산식 기준으로 값을 다시 계산해 미리보기한다. |
| 다시 저장 | 미리보기 값을 같은 주/월 스냅샷에 다시 저장한다. 기존 값은 덮어쓴다. |

- 직접 숫자 수정은 1차 구현에서 하지 않는다.
- 숫자가 틀리면 원천 데이터나 산식을 수정한 뒤 다시 계산한다.
- 다시 저장할 때는 간단한 메모만 받는다. 예: "수술예약 정정 반영", "중단 사유 키워드 보정".
- 이력 테이블은 만들지 않는다. 필요해지면 나중에 감사 로그로 확장한다.

## 6. 화면 표시

- 조회 화면은 지금 그대로 **1벌**. 별도 리포트 조회 페이지는 만들지 않는다.
- 단, 승인/마감 작업을 위한 **마감관리** 메뉴는 추가한다.
- 확정/잠정 구분만 시각화: 표 셀 또는 연도 라벨에 작은 뱃지 — 예) 잠정월은 `*` 또는 옅은 색, 툴팁 "잠정(실시간)".
- 주간 승인값은 `승인`, 월 마감값은 `확정`, 라이브값은 `잠정`으로 표시.
- 당월처럼 잠정인 칸만 표시 다르게, 확정월은 평범하게.

### 6.1 현재 메뉴 분석과 추가 필요 메뉴

현재 메뉴는 지표 조회 중심이다.

```text
HOME
Report
  - 주간 레포트
  - 월간 레포트
전체지표
  - 월별 검사자 종합지표
  - 주간 검사자 종합지표
예약 / 검사 / 전환&성공률 / 수술 / 외래 / 마케팅 / 취소&부도 / 객단가 / 기타
```

주간 승인과 월 마감은 조회 화면 안에 흩뿌리면 운영자가 놓치기 쉽다. 따라서 별도 상위 메뉴가 필요하다.

```text
마감관리
  - 스냅샷 현황
  - 주간 승인
  - 월 마감
  - 재마감/해제
```

권장 화면 역할은 아래와 같다.

| 메뉴 | 목적 | 주요 기능 |
|---|---|---|
| 스냅샷 현황 | 전체 기간의 잠정/승인/확정 상태 확인 | 연월 달력, 주차별 상태, 미승인/미마감 알림, 지표별 누락 표시 |
| 주간 승인 | 주간 리포트 기준값 고정 | 주차 선택, 라이브값 미리보기, 전주 승인값 비교, `승인` 버튼 |
| 월 마감 | 월간 레포트와 월별 통계 기준값 확정 | 승인 주차 합계 vs 월 라이브값 비교, 차이 사유 메모, `월 마감` 버튼 |
| 재마감/해제 | 데이터 정정·규칙 변경 대응 | 승인/확정 해제, 재승인, 재마감, rule_version 확인 |

첫 구현은 `스냅샷 현황 + 주간 승인 + 월 마감` 3개 화면만으로 충분하다. `재마감/해제`는 버튼을 현황 상세에 넣어도 된다.

## 7. 기존 하드코딩 마이그레이션

현재 코드의 레거시 맵을 **seed 데이터**로 한 번에 적재한다.

| 지표 | metric_key | 현재 하드코딩 위치 |
|---|---|---|
| 검사수(전체) | `exam_count` | ProcedureExamStatsService.LEGACY_EXAM_COUNT (2024/2025) |
| 원데이 검사 | `oneday_exam_count` | ProcedureExamStatsService.LEGACY_ONE_DAY_EXAM_COUNT |
| 시력교정검사 | `vision_exam` | ExaminationStatsService.LEGACY_VISION_CORRECTION |
| 백내장검사 | `cataract_exam` | ExaminationStatsService.LEGACY_CATARACT |
| 외래수 | `outpatient_count` | OutpatientCountStatsService.LEGACY_OUTPATIENT_COUNT |

- 적재 후 서비스의 `LEGACY_*` 맵 제거 → 모든 확정값이 테이블 한 곳에서 관리됨.

## 8. 적용 단계 (점진)

1. 월/주간 스냅샷 테이블 + Repository/Service + 읽기 분기(스냅샷 우선) 추가.
2. 기존 레거시 하드코딩을 월 스냅샷 seed로 적재, `LEGACY_*` 맵 제거.
3. `마감관리 > 스냅샷 현황` 화면 추가.
4. 주간 승인 엔드포인트/화면 추가, 주간 레포트·주간 종합지표에 승인값 우선 적용.
5. 월 마감 엔드포인트/화면 추가, 월간 레포트·월별 통계에 확정값 우선 적용.
6. 화면에 `잠정/승인/확정` 뱃지 표시.
7. (선택) 주간 후보 생성·월 자동 마감 스케줄.

## 9. 범위 외 (이번엔 안 함 — 심플 유지)

- **일별 산출물 저장**: 주간 승인까지만 저장한다. 일별은 나중에 필요해지면 확장.
- 스냅샷 이력/버전별 비교 화면, 감사 로그.
- 지표별 단위/소스 메타 테이블화.
- 자동 스케줄(1단계는 수동 승인/마감만).

## 10. 결정 필요

- 주간 기준: 월~일 고정(권장) vs 병원 운영 주차 기준 별도 정의.
- 월 경계 주차 처리: 월 안에서 클립(권장) vs 주 전체를 시작일 월에 귀속.
- 승인/마감 트리거: **수동 버튼**으로 시작(권장) vs 처음부터 스케줄.
- 잠정/승인/확정 표시 방식: 뱃지 / 흐린 색 / 툴팁 중 택1.
- **승인주·확정월 잠금 정책**: 관리자 덮어쓰기·해제 허용(권장, 정정 대응) vs 영구 불변 잠금.
- **이력 보존**: 안 함(덮어쓰면 이전 값 소실, 단순) vs 변경 이력 테이블 추가.

## 11. 프로세스 플로우

### 11.1 조회 흐름 (스냅샷 우선, 없으면 라이브)

```text
weekly query(metricKey + week range)
  |
  v
weekly snapshot exists?
  +-- yes --> approved value   --> UI: 승인
  +-- no  --> live compute     --> UI: 잠정

monthly query(metricKey + year/month range)
  |
  v
monthly snapshot exists?
  +-- yes --> confirmed value  --> UI: 확정
  +-- no  --> live compute     --> UI: 잠정
```

### 11.2 주간 승인 + 월 마감 라이프사이클

```text
[진행주: 라이브 · 잠정]
       |
       |  -- approve-week -->
       v
[승인 주간 스냅샷: 테이블 저장]
       |
       +-- re-approve : 값 덮어쓰기 (정정 · 규칙변경) -> 승인 유지
       |
       +-- delete     : 스냅샷 삭제 -> [진행주: 잠정] 으로 복귀

[월 종료]
       |
       +-- 승인주 합계와 월 라이브값 비교
       |
       +-- close-month -> [확정 월간 스냅샷: 테이블 저장]
```

- **주간 승인(approve-week)**: 진행주 → 라이브 계산 후 주간 스냅샷 저장 → 승인.
- **월 마감(close-month)**: 월 종료 → 승인주 합계와 월 라이브값 비교 후 월간 스냅샷 저장 → 확정.
- **재승인/재마감**: 승인값·확정값 덮어쓰기(값·rule_version·시각 갱신). 데이터 정정·규칙 변경 시.
- **해제(delete)**: 스냅샷 삭제 → 해당 주/월 다시 라이브(잠정)로 복귀.

### 11.3 전체 운영 흐름 (한 달 기준)

```text
[1주차 진행] -- 조회 --> 라이브 계산 (잠정)
       |
       +-- 주 종료 --> 관리자 주간 승인 --> [1주차 승인값 저장]

[2~5주차 반복]
       |
       v
월 종료
       |
       +-- 미승인 주차 있음? -- yes --> 주간 승인 먼저 처리
       |
       +-- no
             |
             v
       승인주 합계 vs 월 라이브값 비교
             |
             v
       관리자 월 마감 --> [월 확정 스냅샷 저장]
             |
             v
       이후 조회 = 고정값 (드리프트 없음)
             |
             v
       정정 필요?
             +-- yes --> 재승인 / 재마감 / 해제 --+
             |                                     |
             +-- no  --> 확정 유지 <--------------+
```
