# 지표 마감 스냅샷 — 기획안 (심플 버전)

> 목적: 가변 소스 테이블(EXAM·RESERVATION 등)에서 나온 월별 지표가 시간이 지나며 미세하게 변하는 문제(드리프트)를 없애고, **리포트 재현성**을 확보한다.
> 원칙: **확정된 달은 저장값(고정), 진행 중인 달은 라이브 계산.** 화면은 1벌 유지.
> 최종 갱신: 2026-06-11 · 상태: 기획(미구현)

---

## 1. 배경

- `EXAM`은 고객당 1행 스냅샷 → 재검사 시 과거월 수치가 사후 변동.
- `RESERVATION` 상태값(`I/H`)은 늦은 내원·정정으로 과거월 행수가 사후 증가.
- 그래서 "그 순간엔 맞던 숫자"가 라이브 재계산 시 ±수 건 어긋난다(외래수 2026-01 레거시 4,822 vs 현재 4,831 등).
- 지금도 2024~2025는 **하드코딩(=수동 스냅샷)** 으로 고정 중. 이걸 테이블로 정식화하는 것이 이 기획의 핵심.

## 2. 핵심 원칙 (단 한 줄)

> **(지표, 연, 월) 스냅샷이 있으면 그 값을 쓰고, 없으면 라이브로 계산한다.**

- 마감된 과거월 → 스냅샷 존재 → **확정값** 표시 (다시는 안 변함)
- 당월/미마감월 → 스냅샷 없음 → **라이브** 계산 (잠정값)

## 3. 테이블 (1개)

운영 앱 DB(H2 개발 / PostgreSQL 운영)에 저장한다. **CRM MSSQL은 READ-ONLY라 쓰지 않는다.**

```sql
CREATE TABLE metric_monthly_snapshot (
  metric_key    VARCHAR(40)  NOT NULL,   -- 예: 'exam_count', 'outpatient_count', 'vision_exam', 'cataract_exam', 'dreamlens_exam'
  year          INT          NOT NULL,
  month         INT          NOT NULL,    -- 1~12
  value         INT,                      -- 확정 수치 (null 허용)
  rule_version  VARCHAR(20)  NOT NULL,    -- 산출 기준 버전 (예: 'v1')
  generated_at  TIMESTAMP    NOT NULL,    -- 확정 시각
  PRIMARY KEY (metric_key, year, month)
);
```

- 컬럼은 이게 전부. 단위·소스 등 메타는 코드/문서가 갖고 있으므로 테이블엔 안 둔다(심플 유지).
- 한 행 = 한 지표의 한 달 확정값.

## 4. 읽기 흐름 (서비스 공통)

```
getMonthly(metricKey, years):
  for each (year, month) in 범위:
    snapshot = snapshotRepo.find(metricKey, year, month)
    value = snapshot != null ? snapshot.value          # 확정
                             : liveCompute(year, month) # 잠정(라이브)
    status = snapshot != null ? 'confirmed' : 'provisional'
```

- 응답 DTO에 `status`(또는 `confirmed: boolean`) 한 필드만 추가.

## 5. 쓰기(마감) 흐름

가장 단순한 형태부터 시작한다.

1. **수동 마감 버튼/엔드포인트** — `POST /api/admin/metric-snapshot/close?ym=2026-05`
   - 해당 월의 모든 지표를 라이브 계산 → `metric_monthly_snapshot`에 upsert (`rule_version`, `generated_at` 기록).
   - 운영자가 "이 달 확정" 누르면 그 값이 고정됨.
2. (선택, 나중에) 스케줄 잡 — 매월 1일 새벽에 전월을 자동 마감.

> 1단계는 수동만으로 충분. 스케줄은 운영 안정화 후 추가.

### 5.1 스냅샷 풀기 (재마감 · 해제)

확정값이 틀렸거나 산출 규칙이 바뀌면 되돌릴 수 있어야 한다. 동작은 2가지뿐.

| 동작 | 방법 | 결과 |
|---|---|---|
| **재마감(덮어쓰기)** | 같은 달 close 다시 호출 | 라이브 재계산 → 같은 행 upsert(`value`·`rule_version`·`generated_at` 갱신). 데이터 정정/규칙 변경 시 사용 |
| **해제(풀기)** | `DELETE /api/admin/metric-snapshot?ym=2026-05` | 스냅샷 행 삭제 → 그 달은 다시 **라이브(잠정)** 로 표시 |

- 둘 다 **관리자 수동**. 확정월을 영구 불변(immutable)으로 잠그지는 않는다 — 현실적으로 정정이 생기기 때문.
- 단순화를 위해 **이력은 남기지 않는다**(덮어쓰면 이전 값은 사라짐). 이력 추적이 필요해지면 그때 별도 테이블로 확장.
- `rule_version`이 바뀌는 전역 규칙 변경 시 = 해당 지표 전체 월을 재마감(일괄 close).

## 6. 화면 표시

- 화면은 지금 그대로 **1벌**. 별도 리포트 페이지 안 만든다.
- 확정/잠정 구분만 시각화: 표 셀 또는 연도 라벨에 작은 뱃지 — 예) 잠정월은 `*` 또는 옅은 색, 툴팁 "잠정(실시간)".
- 당월처럼 잠정인 칸만 표시 다르게, 확정월은 평범하게.

## 7. 기존 하드코딩 마이그레이션

현재 코드의 레거시 맵을 **seed 데이터**로 한 번에 적재한다.

| 지표 | metric_key | 현재 하드코딩 위치 |
|---|---|---|
| 검사수(전체) | `exam_count` | ProcedureExamStatsService.LEGACY_EXAM_COUNT (2024/2025) |
| 원데이 검사 | `oneday_exam_count` | ProcedureExamStatsService.LEGACY_ONE_DAY_EXAM_COUNT |
| 시력교정검사 | `vision_exam` | ExaminationStatsService.LEGACY_VISION_CORRECTION |
| 백내장검사 | `cataract_exam` | ExaminationStatsService.LEGACY_CATARACT |
| 외래수 | `outpatient_count` | OutpatientCountStatsService.LEGACY_OUTPATIENT_COUNT |

- 적재 후 서비스의 `LEGACY_*` 맵 제거 → 모든 확정값이 테이블 한 곳에서 관리됨.

## 8. 적용 단계 (점진)

1. 테이블 + 스냅샷 Repository/Service + 읽기 분기(스냅샷 우선) 추가.
2. 기존 레거시 하드코딩을 seed로 적재, `LEGACY_*` 맵 제거.
3. 수동 마감 엔드포인트 추가, 2026 마감월(1~5월 등) 확정 적재.
4. 화면에 `확정/잠정` 뱃지 표시.
5. (선택) 월 자동 마감 스케줄.

## 9. 범위 외 (이번엔 안 함 — 심플 유지)

- **일별 산출물 저장**(§3): 월별 스냅샷으로 충분. 일별은 나중에 필요해지면 확장.
- 스냅샷 이력/버전별 비교 화면, 감사 로그.
- 지표별 단위/소스 메타 테이블화.
- 자동 스케줄(1단계는 수동 마감만).

## 10. 결정 필요

- 마감 트리거: **수동 버튼**으로 시작(권장) vs 처음부터 스케줄.
- 잠정값 표시 방식: 뱃지 / 흐린 색 / 툴팁 중 택1.
- **확정월 잠금 정책**: 관리자 덮어쓰기·해제 허용(권장, 정정 대응) vs 영구 불변 잠금.
- **이력 보존**: 안 함(덮어쓰면 이전 값 소실, 단순) vs 변경 이력 테이블 추가.

## 11. 프로세스 플로우

### 11.1 조회 흐름 (확정 우선, 없으면 라이브)

```text
월별 지표 조회 (metricKey + 연도범위)
   │
   ▼
각 (연, 월) 스냅샷 있나?
   ├─ 있음 ─▶ 확정값 사용 (confirmed)    ─▶ 화면: 확정 표시
   └─ 없음 ─▶ 라이브 계산 (provisional)  ─▶ 화면: 잠정 뱃지
```

### 11.2 마감 라이프사이클 (생성 · 재마감 · 해제)

```text
[진행월: 라이브 · 잠정]
       │
       │  ── 마감(close) ──▶
       ▼
[확정 스냅샷: 테이블 저장]
       │
       ├─ 재마감(close 재호출) : 값 덮어쓰기 (정정 · 규칙변경)  ↺ 확정 유지
       │
       └─ 해제(delete) : 스냅샷 삭제  ──▶ [진행월: 잠정] 으로 복귀
```

- **마감(close)**: 진행월 → 라이브 계산 후 스냅샷 저장 → 확정.
- **재마감(close 재호출)**: 확정값 덮어쓰기(값·rule_version·generated_at 갱신). 데이터 정정·규칙 변경 시.
- **해제(delete)**: 스냅샷 삭제 → 그 달 다시 라이브(잠정)로 복귀.

### 11.3 전체 운영 흐름 (한 달 기준)

```text
[당월 진행 중] ──매일 조회──▶ 라이브 계산 (잠정 표시)
       │
       ▼
   월 종료?
       ├─ 아니오 ──▶ 계속 라이브 (잠정)
       └─ 예 ──▶ 관리자 마감 ──▶ [확정 스냅샷 저장]
                                      │
                                      ▼
                          이후 조회 = 고정값 (드리프트 없음)
                                      │
                                      ▼
                                 정정 필요?
                                      ├─ 규칙변경 · 오류 ──▶ 재마감 / 해제 ──┐
                                      └─ 아니오 ──▶ 확정 유지                │
                                            ▲────────────────────────────────┘
```
