# 지표 정의 (검사 · 수술 · 예약)

> 대시보드 집계 지표의 **공식 정의서**. 병원 자체 집계와 수치가 어긋날 때 이 문서가 기준이다.
> 모든 SQL은 MSSQL(SOFTCRM) 기준, READ-ONLY, 대용량 테이블은 `WITH(NOLOCK)` + 날짜 범위 필수.
> 최종 갱신: 2026-06-11

## 정의 상태

| 지표 | 상태 | 비고 |
|------|------|------|
| 검사건수 | ✅ 정의됨 | 시력교정 · 백내장 · 드림렌즈 3종 |
| 검사수(전체) | ✅ 정의됨 | `검사건수` 메뉴 화면 — EXAM 행수 + Cataract_Exam 세션수 raw 합산 (§1.9) |
| 수술건수 | ✅ 정의됨 | 실제 수술일 기준 · 시력교정 검수값 일부 고정 |
| 일별 산출물 | ✅ 운영원칙 정의 | 2026년 이후 일별 저장, 월/주/기간은 합산 |
| 외래수 | ✅ 정의됨 | 2024~2025 레거시 고정, 2026~ DB(`F`+`I/H` 행수) (§4) |
| 예약건수 | ⬜ 미정의 | TODO |

## 1. 검사건수

검사는 **시력교정 · 백내장 · 드림렌즈** 3종으로 구분한다. **각 카테고리는 소스·집계단위가 서로 다르다.**

### 1.1 개요

| 카테고리 | 소스 테이블 | 분류 조건 | 집계 단위 | 대표 검증값 |
|---|---|---|---|---|
| **시력교정검사** | `EXAM` (검사자 리스트) | `EXAM_DATE` 기준, **드림렌즈 분류 제외** (측정값 무관) | **행/사람** | 2026-04 = 809 |
| **백내장검사** | `Cataract_Exam` | 같은 날 백내장 검사예약 `H` 내원 + 추천 수술방법 입력 | **눈** (좌/우 각각) | 2026-01~04 = 96 / 66 / 64 / 58 |
| **드림렌즈검사** | `EXAM` (검사자 리스트) | 같은 검사일 렌즈센터 `D`만 있고 검사 `M`이 없는 행 | **행/사람** | 2026-04 = 17 |

> ⚠️ **단위가 달라 단순 합산은 의미가 다르다** (시력교정·드림렌즈=사람, 백내장=눈). 통합 "총 검사건수"가 필요하면 단위를 통일하는 별도 합의가 필요하다.

### 1.2 공통 규칙

**검사결과 테이블 기반** (시력교정=`EXAM`, 백내장=`Cataract_Exam`)
- 날짜 기준: `EXAM_DATE` (char(10), `YYYY-MM-DD`) — 실제 검사일
- **백내장은 "추천 수술방법 존재"(`OPERATIONR`/`OPERATIONL`)로 유효 눈 판정**. 같은 날 백내장 검사예약(`RESERVE_FLAG='H'`) 내원 상태가 있어야 한다.
- **시력교정은 측정값 무관** — 병원 검사자 리스트 행수 기준이라 측정값 미입력 검사도 포함.
- 더 엄격한 "완료" 기준이 필요하면 `STOP_YN='Y'`(중단)·`CANCEL_CD`(취소)를 추가 제외.

**EXAM + RESERVATION 분류 기반** (드림렌즈)
- 소스: `EXAM WITH(NOLOCK)` 기준 행을 `RESERVATION` 렌즈센터 `D` 예약으로 분류
- 상태: `RESERVE_STATE IN ('I','H')` — 예약(Y)·취소(C) 제외. 이 병원은 'I'에서 멈추므로 사실상 **"내원(체크인)"**.
- 날짜 기준: `RESERVE_DATE`
- 정기검진(FU)은 제외.

### 1.3 시력교정검사

**정의**: 해당 기간에 시력교정 검사를 실제 받은 **사람 수**. 소스는 검사결과 테이블 `EXAM` (병원 "검사자 리스트(상담사별)" 화면과 동일 소스). **EXAM에 섞인 드림렌즈는 제외.**

| 항목 | 값 |
|------|-----|
| 소스 | **`EXAM`** (시력교정 검사결과, PK=`CUST_NUM`) |
| 날짜 기준 | `EXAM_DATE` (char(10), `YYYY-MM-DD`) |
| 측정값 필터 | **없음** — 병원 검사자 리스트 행수 기준(측정값 미입력 검사도 포함). 엄격 "실시"분만 필요하면 `RIGHT01`/`LEFT01` 존재 추가 |
| 드림렌즈 제외 | 같은 날 렌즈센터(`RESERVE_FLAG='D'`) 예약만 있고 시력교정(`'M'`) 예약 없는 EXAM 배제 |
| 집계 단위 | **사람** = `COUNT(DISTINCT CUST_NUM)` (EXAM은 고객당 1행이라 행수와 동일) |

> **소스가 RESERVATION이 아니라 `EXAM`인 이유**: 병원 공식 "검사자 리스트(상담사별)" 화면(`FrmCounselExamList`)이 시력교정을 `EXAM`+`EXAM_DATE`로 집계한다. "실제 검사 실시" 기준이며 백내장(`Cataract_Exam`)과 동일 구조다. RESERVATION(flag M) 예약기반 집계는 검사 실시와 불일치(과거 정의 843/805 → 본 정의로 교체).
> **드림렌즈 혼입 주의**: `EXAM`에는 검사종류 구분 컬럼이 없어 드림렌즈(CRT/각막굴절교정) 검사가 섞여 있다. 같은 날 렌즈센터(`D`) 예약 기준으로 배제한다.

```sql
SELECT SUBSTRING(e.EXAM_DATE,1,7) AS ym,
       COUNT(DISTINCT e.CUST_NUM) AS visionCorrection
FROM EXAM e WITH(NOLOCK)
WHERE e.EXAM_DATE >= :from AND e.EXAM_DATE <= :to
  AND NOT (                                    -- 드림렌즈 배제
        EXISTS (SELECT 1 FROM RESERVATION rd WITH(NOLOCK)
                WHERE rd.CUST_NUM = e.CUST_NUM AND rd.RESERVE_DATE = e.EXAM_DATE
                  AND rd.RESERVE_STATE IN ('I','H') AND rd.RESERVE_FLAG = 'D')
    AND NOT EXISTS (SELECT 1 FROM RESERVATION rm WITH(NOLOCK)
                WHERE rm.CUST_NUM = e.CUST_NUM AND rm.RESERVE_DATE = e.EXAM_DATE
                  AND rm.RESERVE_STATE IN ('I','H') AND rm.RESERVE_FLAG = 'M')
  )
GROUP BY SUBSTRING(e.EXAM_DATE,1,7)
ORDER BY ym
```

- 보조(엄격 "실시" 기준): 위 조건 + 측정값 존재(`RIGHT01`/`LEFT01`) → 측정 완료분만 02=1585 / 03=850 / 04=799 (데스크는 미입력도 세므로 기본 정의는 측정값 무관).
- **검증 (병원 수기 대조)**: 02=1600(병원 1604) / 03=854(853) / 04=809(807) / 05=900. 04월은 병원 엑셀 826행 = EXAM 4월 전체와 1:1 대조로 확정. 드림렌즈 미제외 시 02=1611/03=873/04=826.

### 1.4 백내장검사

**정의**: 해당 기간에 백내장 검사를 받은 뒤 **추천 수술방법이 기입된 눈(좌/우)** 을 집계. 양안 추천 = 2건, 단안 추천 = 1건.

| 항목 | 값 |
|------|-----|
| 소스 | **`Cataract_Exam`** (실제 백내장 검사 상세, `CUST_NUM`+`SEQ`) |
| 날짜 기준 | `EXAM_DATE` (char(10), `YYYY-MM-DD`) |
| 예약 보조조건 | 같은 고객·같은 날짜 `RESERVATION.RESERVE_FLAG='H'`, `RESERVE_STATE IN ('I','H')` 존재 |
| 제외 | `Stop_YN='Y'`(중단), `Cancel_CD` 채워짐(취소), 테스트 고객명 |
| 눈 판정 | 우안 = `OPERATIONR` 채워짐, 좌안 = `OPERATIONL` 채워짐 |
| 집계 단위 | **눈 (좌/우 각각 1건)** = `SUM(OPERATIONR 채움) + SUM(OPERATIONL 채움)` |

> **측정값(`RIGHT01`/`LEFT01`) 기준을 사용하지 않음.** 레거시 백내장 검사건수와 대조한 결과, 단순 측정 눈 수(2026-04=214안)가 아니라 **백내장 수술 추천을 받은 눈 수**가 운영 지표에 맞는다.
> 2024~2025년은 레거시 확정값을 고정 사용하고, 2026년 이후는 아래 기준으로 계산한다.

```sql
SELECT SUBSTRING(ce.EXAM_DATE,1,7) AS ym,
       SUM(CASE WHEN NULLIF(LTRIM(RTRIM(ISNULL(ce.OPERATIONR,''))), '') IS NOT NULL THEN 1 ELSE 0 END)
     + SUM(CASE WHEN NULLIF(LTRIM(RTRIM(ISNULL(ce.OPERATIONL,''))), '') IS NOT NULL THEN 1 ELSE 0 END) AS cataract
FROM Cataract_Exam ce WITH(NOLOCK)
JOIN CUSTOM cu WITH(NOLOCK) ON ce.CUST_NUM = cu.CUST_NUM
WHERE ce.EXAM_DATE >= :from AND ce.EXAM_DATE <= :to
  AND (ce.Stop_YN IS NULL OR ce.Stop_YN <> 'Y')
  AND (ce.Cancel_CD IS NULL OR LTRIM(RTRIM(ce.Cancel_CD)) = '')
  AND NOT (ISNULL(cu.CUST_NAME,'') LIKE N'%테스트%' OR LOWER(ISNULL(cu.CUST_NAME,'')) LIKE '%test%')
  AND EXISTS (
    SELECT 1
    FROM RESERVATION r WITH(NOLOCK)
    WHERE r.CUST_NUM = ce.CUST_NUM
      AND r.RESERVE_DATE = ce.EXAM_DATE
      AND r.RESERVE_FLAG = 'H'
      AND r.RESERVE_STATE IN ('I','H')
  )
GROUP BY SUBSTRING(ce.EXAM_DATE,1,7)
ORDER BY ym
```

- 보조: 검사인원 = `COUNT(DISTINCT ce.CUST_NUM)`, 검사세션 = `COUNT(*)`, 측정 눈 수는 품질 점검용으로만 사용.
- **2026 검증값**: 01=96 / 02=66 / 03=64 / 04=58.
- **레거시 고정값**: 2024~2025는 화면 표의 숫자를 그대로 사용. 2026은 DB 계산값 사용.

### 1.4.1 백내장예약률

**정의**: 백내장 검사건수로 선별된 추천 눈(안) 중, 백내장 수술예약으로 전환된 비율.

| 항목 | 값 |
|------|-----|
| 화면명 | `백내장예약률 (수술예약건/검사자)` |
| 분모 | `백내장검사`와 동일한 추천 눈 수 (`Cataract_Exam.OPERATIONR/L`) |
| 분자 | 분모에 포함된 고객 중 취소 아님인 수술예약 `RESERVATION.RESERVE_FLAG='O'` 존재 |
| 날짜 기준 | 분모는 `Cataract_Exam.EXAM_DATE`, 수술예약은 검사자 리스트의 `수술예약등록일` 존재 여부와 동일하게 판정 |
| 집계 단위 | 분모는 **눈(안)**, 분자는 **고객/예약 건** |
| 2024~2025 | 레거시 예약률 표 숫자 고정 사용 |
| 2026 이후 | DB 계산값 사용 |

> **검사 예약등록일은 분자 아님.** 백내장 검사건수 분모는 이미 같은 날 백내장 검사예약(`H`) 내원을 보조조건으로 요구하므로, 검사예약 존재 여부를 분자로 쓰면 사실상 100%가 된다. 예약률 분자는 반드시 **수술예약(`O`)** 이어야 한다.
> `RESERVATION`에는 좌/우 눈 구분이 없으므로 분자는 눈 단위로 증폭하지 않고, 검사자 리스트의 `수술예약등록일` 존재 여부처럼 고객/예약 건 단위로 집계한다.

### 1.5 드림렌즈검사

**정의**: 해당 기간 드림렌즈/렌즈센터 검사 건수. 검사자 리스트와 같은 `EXAM` 기준에서 같은 날 렌즈센터 `D` 예약만 있고 검사 `M` 예약이 없는 행을 집계.

| 항목 | 값 |
|------|-----|
| 소스 | `EXAM` + `RESERVATION` 분류 |
| 진료유형 | `RESERVE_FLAG = 'D'` |
| 상태 | `RESERVE_STATE IN ('I','H')` |
| 세부유형 | 제한 없음 |
| 집계 단위 | **`COUNT(DISTINCT CUST_NUM)` (사람단위)** — 시력교정과 통일 |

> 2026-02만 보고 D/L 계열(`1`,`5`,`7`)로 좁히면 01·03·04월 시력교정 숫자가 더 흔들림. 현재는 원복하여 렌즈센터 `D` 전체를 드림렌즈 분류로 사용.

```sql
SELECT SUBSTRING(e.EXAM_DATE,1,7) AS ym,
       COUNT(*) AS dreamlensPersons
FROM EXAM e WITH(NOLOCK)
WHERE e.EXAM_DATE >= :from AND e.EXAM_DATE <= :to
  AND EXISTS (SELECT 1
              FROM RESERVATION rd WITH(NOLOCK)
              WHERE rd.CUST_NUM = e.CUST_NUM
                AND rd.RESERVE_DATE = e.EXAM_DATE
                AND rd.RESERVE_STATE IN ('I','H')
                AND rd.RESERVE_FLAG = 'D')
  AND NOT EXISTS (SELECT 1
                  FROM RESERVATION rm WITH(NOLOCK)
                  WHERE rm.CUST_NUM = e.CUST_NUM
                    AND rm.RESERVE_DATE = e.EXAM_DATE
                    AND rm.RESERVE_STATE IN ('I','H')
                    AND rm.RESERVE_FLAG = 'M')
GROUP BY SUBSTRING(e.EXAM_DATE,1,7)
ORDER BY ym
```

- **검증**: 02=11 / 03=19 / 04=17 / 05=23 명. 검사자 리스트 전체에서 분리한 값.

### 1.6 함정 (Gotchas)

- **카테고리별 소스·단위 상이**: 시력교정=`EXAM`/사람, 백내장=`Cataract_Exam`/추천 눈, 드림렌즈=`EXAM`+`RESERVATION` 분류/사람. 코드·차트에 소스/단위 라벨 명시 필수.
- **시력교정 소스 주의**: 예약(`RESERVATION` flag M)이 아니라 **검사결과(`EXAM`)** 기준. 예약기반은 병원 공식 수치와 어긋난다.
- **EXAM에 드림렌즈 혼입**: `EXAM`엔 검사종류 구분 컬럼이 없어 드림렌즈(CRT/각막굴절교정) 검사가 월 ~2%(11-21명) 섞인다. 같은 날 렌즈센터(`D`) 예약 기준으로 반드시 배제(미배제 873/826 → 배제 854/809).
- **측정값 필터는 시력교정 기본 정의에서 미적용**: 병원 데스크가 측정값 미입력 검사도 리스트에 세므로 행수 기준으로 맞춘다. 측정값 존재(`RIGHT01`/`LEFT01`) 조건은 엄격 "실시" 보조지표로만(측정분 1585/850/799). 백내장은 측정값이 아니라 추천 수술방법(`OPERATIONR/L`) 기준.
- **EXAM은 스냅샷(고객당 1행) → 과거 월 수치가 사후 감소**: 고객이 나중에 재검사하면 `EXAM_DATE`가 덮어써져 원래 달에서 빠진다. 실측 월 -3~-5명(~0.3%) 수준. 과거 월 수치는 조회 시점에 따라 미세하게 달라질 수 있음.
- **측정값 미입력 검사(월 ~12명)**: `EXAM_DATE`는 해당월인데 `RIGHT01`/`LEFT01`이 빈 검사. 병원 데스크와 본 정의(측정값 무관)는 포함, 엄격 "실시" 보조지표에선 제외.
- **백내장 단위 혼동 금지**: "추천 눈" 단위라 인원과 다를 수 있음. 시력교정·드림렌즈(사람)와 직접 비교 시 단위 라벨을 반드시 확인.
- **드림렌즈 분류 기준 주의**: 02월만 보면 D/L 계열만 분리하는 기준이 맞아 보이나, 01·03·04월은 렌즈센터 `D` 전체 분리가 레거시 시력교정 수치에 더 가깝다.
- **드림렌즈 상태값 `I` 단독**: `RESERVATION` `IN('I','H')`는 사실상 "내원(체크인)"이며, 최종 집계는 `EXAM` 행 존재를 함께 요구한다.
- **병원 수기집계와 ±오차 정상**: 병원 자체 수치(시력교정 03=853/04=807 등)는 수기/엑셀 집계라 시스템 정의와 1~2% 차이는 자연스럽다.

### 1.7 검증 이력 (2026년)

| 카테고리 | 단위 | 2026-01 | 2026-02 | 2026-03 | 2026-04 |
|---|---|--:|--:|--:|--:|
| 시력교정검사 | 사람 (EXAM, 드림렌즈 분류 제외) | 1,873 | 1,600 | 854 | 809 |
| 백내장검사 | 추천 눈 (좌/우) | 96 | 66 | 64 | 58 |

#### 1.9.1 검사수

**정의**: 레거시 "검사수" 차트(병원 엑셀)를 재현한 **전체 검사 건수**.
`검사수 = EXAM 검사자 행수 + Cataract_Exam 검사 세션수`. **둘 다 raw `COUNT(*)`** — 테스트/취소/드림렌즈/눈 단위 필터를 적용하지 않는다.

| 구성 | 소스 | 집계 |
|---|---|---|
| EXAM 행수 | `EXAM` | `COUNT(*)` (시력교정 + 드림렌즈 = 검사자 리스트 전체 행) |
| Cataract_Exam 세션수 | `Cataract_Exam` | `COUNT(*)` (모든 검사행, 눈 단위 아님) |

```sql
SELECT t.yr, t.mo, SUM(t.cnt) AS cnt
FROM (
    SELECT CAST(SUBSTRING(EXAM_DATE,1,4) AS INT) AS yr, CAST(SUBSTRING(EXAM_DATE,6,2) AS INT) AS mo, COUNT(*) AS cnt
    FROM EXAM WITH(NOLOCK) WHERE EXAM_DATE >= :from AND EXAM_DATE <= :to
    GROUP BY CAST(SUBSTRING(EXAM_DATE,1,4) AS INT), CAST(SUBSTRING(EXAM_DATE,6,2) AS INT)
    UNION ALL
    SELECT CAST(SUBSTRING(EXAM_DATE,1,4) AS INT) AS yr, CAST(SUBSTRING(EXAM_DATE,6,2) AS INT) AS mo, COUNT(*) AS cnt
    FROM Cataract_Exam WITH(NOLOCK) WHERE EXAM_DATE >= :from AND EXAM_DATE <= :to
    GROUP BY CAST(SUBSTRING(EXAM_DATE,1,4) AS INT), CAST(SUBSTRING(EXAM_DATE,6,2) AS INT)
) t
GROUP BY t.yr, t.mo ORDER BY t.yr, t.mo
```

> **§1 카테고리 정의와의 차이 (중요)**: §1의 화면 `전체`(`examTotal`)는 백내장을 **추천 눈(96)** 으로 세지만, 본 검사수는 백내장을 **Cataract_Exam raw 세션(145)** 으로 센다. 시력교정+드림렌즈 부분(EXAM 전체 행)은 양쪽 동일.
> 예) 2026-01: §1 `examTotal` = 1,873 + 20 + 96 = **1,989** vs 검사수 = 1,893 + 145 = **2,038** (차이 49 = 145 − 96).

#### 1.9.2 원데이 검사

**정의**: 레거시 "원데이 검사" 차트를 재현하기 위한 검사OP 기반 검사 건수. 실제 같은 날 수술 여부가 아니라, 검사자 리스트(`EXAM`) 중 같은 날 `RESERVATION` 시력교정 검사예약 `M/5(검사OP)`로 내원한 건을 센다.

| 항목 | 값 |
|---|---|
| 소스 | `EXAM` + `RESERVATION` |
| 날짜 기준 | `EXAM.EXAM_DATE` |
| 원데이 판정 | 같은 고객·같은 날짜 `RESERVATION.RESERVE_FLAG='M'`, `RESERVE_JINRYO='5'`, `RESERVE_STATE IN ('I','H')` 존재 |
| 제외 | `EXAM.STOP_YN='Y'`, `EXAM.CANCEL_CD` 존재, 테스트 고객명/검사메모/예약번호 |
| 집계 단위 | `COUNT(*)` (EXAM 행/사람 단위) |

```sql
SELECT SUBSTRING(e.EXAM_DATE,1,7) AS ym,
       COUNT(*) AS oneDayExamCount
FROM EXAM e WITH(NOLOCK)
JOIN CUSTOM cu WITH(NOLOCK) ON cu.CUST_NUM = e.CUST_NUM
WHERE e.EXAM_DATE >= :from AND e.EXAM_DATE <= :to
  AND ISNULL(e.STOP_YN, '') <> 'Y'
  AND ISNULL(e.CANCEL_CD, '') = ''
  AND NOT (
    ISNULL(cu.CUST_NAME, '') LIKE N'%테스트%'
    OR LOWER(ISNULL(cu.CUST_NAME, '')) LIKE '%test%'
    OR ISNULL(e.EXAM_MEMO, '') LIKE N'%테스트%'
    OR LOWER(ISNULL(e.EXAM_MEMO, '')) LIKE '%test%'
  )
  AND EXISTS (
    SELECT 1
    FROM RESERVATION r WITH(NOLOCK)
    WHERE r.CUST_NUM = e.CUST_NUM
      AND r.RESERVE_DATE = e.EXAM_DATE
      AND r.RESERVE_FLAG = 'M'
      AND r.RESERVE_STATE IN ('I','H')
      AND r.RESERVE_JINRYO = '5'
      AND LOWER(ISNULL(r.RESERVE_NUM, '')) NOT LIKE '%test%'
      AND LOWER(ISNULL(r.RESERVE_NUM, '')) NOT LIKE '%kiosktest%'
  )
GROUP BY SUBSTRING(e.EXAM_DATE,1,7)
ORDER BY ym
```

## 2. 수술건수

수술은 **실제 수술일자** 기준으로 집계한다. 수술예약일/예약등록일이 아니라 실제 수술 실행 테이블을 기준으로 함.

### 2.1 개요

| 카테고리 | 소스 테이블 | 날짜 기준 | 집계 단위 | 비고 |
|---|---|---|---|---|
| **시력교정수술** | `OPERATIONDATA` | `OPERATION_DATE` | **사람** (`DISTINCT CUST_NUM`) | 백내장 수술 고객 제외 |
| **백내장수술** | `Cataract_Operationdata` | `OPERATIONR_DATE` / `OPERATIONL_DATE` | **눈(안)** (`CUST_NUM + 수술일 + R/L`) | 레거시 백내장 수술건수 기준 |

### 2.2 공통 규칙

- 테스트 고객 제외: 고객명 `테스트`, `test` 포함 제외.
- 무효 수술코드 제외: `X`, `OP불가`, `모든수술가능`, `op x`, `Strabismus`, `TEST-TEST`.
- 시력교정은 `Cataract_Operationdata`에 존재하는 고객을 제외하여 백내장 중복 혼입 방지.
- `전체 수술건수`는 세부 수술명 합산이 아니라 `시력교정수술 인원 + 백내장수술 눈 수`.
- 세부 수술명은 양안에 서로 다른 수술명이 들어갈 수 있어 유형별 합계와 전체 인원이 반드시 같지 않음.

### 2.3 시력교정수술

**정의**: 해당 기간 실제 시력교정 수술을 받은 사람 수. 레거시 "시력교정 검사 후 수술" 표와 대조한 결과, 수술예약보다 `OPERATIONDATA.OPERATION_DATE` 실제 수술 기준에 가장 가까움.

```sql
SELECT SUBSTRING(o.OPERATION_DATE,1,7) AS ym,
       COUNT(DISTINCT o.CUST_NUM) AS visionPatients
FROM OPERATIONDATA o WITH(NOLOCK)
JOIN CUSTOM cu WITH(NOLOCK) ON cu.CUST_NUM = o.CUST_NUM
WHERE o.OPERATION_DATE >= :from AND o.OPERATION_DATE <= :to
  AND NOT EXISTS (
      SELECT 1
      FROM Cataract_Operationdata co WITH(NOLOCK)
      WHERE co.CUST_NUM = o.CUST_NUM
  )
  AND (
       (NULLIF(LTRIM(RTRIM(ISNULL(o.OPERATIONR,''))), '') IS NOT NULL
        AND o.OPERATIONR NOT IN ('X','OP불가','모든수술가능','op x','Strabismus','TEST-TEST'))
    OR (NULLIF(LTRIM(RTRIM(ISNULL(o.OPERATIONL,''))), '') IS NOT NULL
        AND o.OPERATIONL NOT IN ('X','OP불가','모든수술가능','op x','Strabismus','TEST-TEST'))
  )
  AND NOT (
      ISNULL(cu.CUST_NAME, '') LIKE N'%테스트%'
      OR LOWER(ISNULL(cu.CUST_NAME, '')) LIKE '%test%'
  )
GROUP BY SUBSTRING(o.OPERATION_DATE,1,7)
ORDER BY ym
```

### 2.4 백내장수술

**정의**: 해당 기간 실제 백내장 수술 눈 수. 우/좌 수술일을 펼친 뒤 `CUST_NUM + OPERATION_DATE + R/L`을 1건으로 집계. 동일인이 같은 날 양안 수술하면 2안으로 집계.

```sql
SELECT SUBSTRING(x.OPERATION_DATE,1,7) AS ym,
       COUNT(DISTINCT x.CUST_NUM + '|' + x.OPERATION_DATE + '|' + x.eye) AS cataractPatients
FROM (
    SELECT CUST_NUM, OPERATIONR_DATE AS OPERATION_DATE, OPERATIONR AS OPERATION, 'R' AS eye
    FROM Cataract_Operationdata WITH(NOLOCK)
    UNION ALL
    SELECT CUST_NUM, OPERATIONL_DATE AS OPERATION_DATE, OPERATIONL AS OPERATION, 'L' AS eye
    FROM Cataract_Operationdata WITH(NOLOCK)
) x
JOIN CUSTOM cu WITH(NOLOCK) ON cu.CUST_NUM = x.CUST_NUM
WHERE x.OPERATION_DATE >= :from AND x.OPERATION_DATE <= :to
  AND NULLIF(LTRIM(RTRIM(ISNULL(x.OPERATION,''))), '') IS NOT NULL
  AND x.OPERATION NOT IN ('X','OP불가','TEST-TEST')
  AND NOT (
      ISNULL(cu.CUST_NAME, '') LIKE N'%테스트%'
      OR LOWER(ISNULL(cu.CUST_NAME, '')) LIKE '%test%'
  )
GROUP BY SUBSTRING(x.OPERATION_DATE,1,7)
ORDER BY ym
```

- [SurgeryStatsService.java](../../backend/src/main/java/com/bviit/analytics/service/stats/SurgeryStatsService.java):
  - 2024~2025년 1~4월, 2026년 1~4월 시력교정수술 레거시 검수값 고정 ✅
  - 기본 JSON 캐시 사용. 캐시 파일이 없으면 DB 쿼리 후 `.cache/stats/surgery-monthly-v3-*.json` 저장.
- `/stats/surgery` 화면의 `전체 수술건수`는 세부 유형 합산이 아니라 `visionPatients + cataractPatients`.

## 3. 산출물 저장 전략

대시보드 지표는 **2026년 이후 데이터부터 일별 산출물 저장 → 월/주/기간 화면은 일별 합산**을 원칙으로 한다.

### 3.1 기준

| 기간 | 저장/집계 방식 | 이유 |
|---|---|---|
| **2024~2025년** | 레거시 월별 확정값 사용 | 일별 원본 검수값이 없고, 병원 공식 월별 값이 기준 |
| **2026년 이후** | 일별 산출물 저장 후 합산 | 주차/월/사용자 기간 검색을 같은 기준으로 처리하고 차이 원인 추적 가능 |

### 3.2 대상 지표

Airflow 계열 함수명 기준으로 아래 6개 일별 산출물을 우선 대상으로 둔다.

| 센터 | 검사 | 수술 | 매출 |
|---|---|---|---|
| **시력교정센터** | `get_daily_exam_count_result` | `get_daily_operation_count_result` | `get_daily_revenue_result` |
| **백내장** | `get_daily_cataract_exam_count_result` | `get_daily_cataract_operation_count_result` | `get_daily_cataract_revenue_result` |

### 3.3 저장 단위

권장 저장 형태:

```text
daily_metric_result
- metric_date       -- YYYY-MM-DD
- center_type       -- vision | cataract
- metric_type       -- exam_count | operation_count | revenue
- value             -- 산출값
- unit              -- person | eye | won
- rule_version      -- 산출 기준 버전
- generated_at      -- 산출 시각
```

운영 원칙:

- 화면의 월별 값은 `metric_date` 월 범위 합산으로 만든다.
- 화면의 주차/퀵버튼/사용자 기간 값도 동일한 일별 산출물을 합산한다.
- 2024~2025년 레거시 고정값은 월별 기준값으로만 보관한다. 임의 일별 배분 금지.
- 2026년 이후 일별 산출물은 산출 기준 변경 시 `rule_version`을 올려 재생성한다.
- `EXAM`처럼 사후 변경되는 스냅샷 테이블은 일별 산출물을 저장해 월별 숫자 흔들림을 추적한다.

### 3.4 현재 코드와의 관계

현재 Java Dashboard는 검사/수술을 **월별 직접 집계 + JSON 캐시**로 처리한다.

| 지표 | 현재 구현 | 향후 방향 |
|---|---|---|
| 검사건수 | 월별 쿼리 + `.cache/stats/examination-monthly-v7-*` | 2026년 이후 일별 검사 산출물 저장 후 합산 |
| 수술건수 | 월별 쿼리 + `.cache/stats/surgery-monthly-v3-*` | 2026년 이후 일별 수술 산출물 저장 후 합산 |
| 매출 | B2B 매출 일부 구현 | 일반 시력교정/백내장 일별 매출 기준 별도 정의 필요 |

## 4. 외래수

- **2024~2025**: 레거시 차트 확정값 고정.
- **2026년 이후**: 운영 DB 산출값. 외래 예약(`RESERVE_FLAG='F'`) 중 내원/퇴원 처리된(`RESERVE_STATE IN ('I','H')`) **행 수**(`COUNT(*)`, 사람 단위 아님).

### 4.1 정의

| 항목 | 값 |
|---|---|
| 화면 | `/stats/outpatient-count` |
| 메뉴 | `외래 > 외래수` |
| 소스 | `RESERVATION` (`RESERVE_FLAG='F'`, `RESERVE_STATE IN ('I','H')`) |
| 날짜 기준 | `RESERVE_DATE` (char(10), `YYYY-MM-DD`) |
| 집계 단위 | **행 수** (`COUNT(*)`) |
| 2024~2025 | 레거시 확정값 고정 (DB 재집계 안 함) |
| 2026 이후 | DB 라이브 계산 (mssql 프로파일) |

```sql
SELECT SUBSTRING(r.RESERVE_DATE,1,7) AS ym, COUNT(*) AS cnt
FROM RESERVATION r WITH(NOLOCK)
WHERE r.RESERVE_DATE >= :from AND r.RESERVE_DATE <= :to
  AND r.RESERVE_FLAG = 'F'
  AND r.RESERVE_STATE IN ('I','H')
GROUP BY SUBSTRING(r.RESERVE_DATE,1,7)
ORDER BY ym
```

### 4.2 값

**2024~2025 레거시 확정값**

| 연도 | 01 | 02 | 03 | 04 | 05 | 06 | 07 | 08 | 09 | 10 | 11 | 12 |
|---:|--:|--:|--:|--:|--:|--:|--:|--:|--:|--:|--:|--:|
| 2024 | 5257 | 5080 | 4721 | 4133 | 4159 | 4092 | 4760 | 4720 | 3696 | 4201 | 3948 | 4711 |
| 2025 | 5389 | 5389 | 4933 | 4349 | 4451 | 4000 | 4275 | 4354 | 3332 | 3724 | 3639 | 4086 |

**2026 DB 검증 (운영 DB 실계산, 5월 이후는 라이브)**

| 월 | 레거시(참고) | DB 계산 | 차이 |
|---|--:|--:|--:|
| 01 | 4822 | 4831 | +9 |
| 02 | 4339 | 4344 | +5 |
| 03 | 4247 | 4251 | +4 |
| 04 | 3653 | 3658 | +5 |

> DB 계산이 레거시보다 월 4~9건 높다. 테스트 고객/예약번호/메모를 제외해도 거의 동일 → 레거시 출력 시점 이후 상태값 변경 또는 일부 외래 세부구분 차이로 추정. **레거시가 아니라 DB 라이브 값을 공식 표시값으로 사용**한다.

### 4.3 함정 (Gotchas)

- **2025-02 세부구분 이슈**: 레거시 5,389 vs DB 5,673 (+284). `RESERVE_JINRYO='3'`이 294건이라 거의 원인처럼 보이나, 이 코드를 전역에서 제외하면 다른 월이 크게 틀어진다 → **전역 규칙으로 쓰지 말 것**. 2024~2025는 레거시 고정이므로 이 차이는 표시에 영향 없음.
- **사람 수 아님**: 행 수 기준이라 동일인 복수 외래 예약이 각각 1건으로 잡힌다.
